PumpSDK — MCP Server

Launch, trade, and manage Solana tokens on Pump.fun through 36 Model Context Protocol tools. Zero-custody signing, AI-powered scouting, real-time streams, and an autonomous agent system — all from your MCP client.

36 Tools 4 Resources 6 Prompts npm pumpsdk GitHub

Key Features

Zero-Custody Trading Build unsigned transactions server-side. Sign in your own browser wallet. No private keys ever leave your machine.
Token Launching Multi-phase launch flow with IPFS metadata upload, dev buy slider, fee config, and 8-wallet signing support.
AI Scout Scan trends, generate token art with Nano Banana 2 Pro, and launch — all in a single guided flow.
Real-Time Streams WebSocket subscriptions for new mints, trade events, and graduation monitoring via PumpPortal.
Fee Management Compose fee configs, claim creator fees, batch claim across tokens, and register partner splits.
Agent System Dual-model orchestrator with 6 autonomous strategies: sniping, scouting, fee optimization, and more.
Analytics Bonding curve analysis, holder distribution, DexScreener integration, and top-token leaderboards.
Nano Banana Image Gen Creative-director prompt builder with style lineages, palette pairs, and abstract/concrete detection.

Who Is This For

  • MCP Client Users — Claude Desktop, Cursor, or any MCP-compatible client that connects via stdio or HTTP
  • Token Creators — launch tokens on Pump.fun with metadata, images, fee splits, and dev buys
  • Traders — get quotes, execute swaps, and monitor positions through natural language
  • Developers — extend the server with custom tools, strategies, or transport layers
  • Autonomous Agents — run 24/7 with strategies that claim fees, snipe launches, and monitor graduations

Architecture

How PumpSDK is structured internally.

Layer Overview

PumpSDK follows a layered architecture with clear separation between transport, tools, clients, and signing:

  • Transport Layer — stdio (default) or HTTP/SSE for remote connections
  • MCP Server — registers 36 tools, 4 resources, and 6 prompts on a single McpServer instance
  • Tool Layer — 9 domain groups (trading, tokens, analytics, fees, scout, solana, state, stream, meta)
  • Client Layer — REST clients for Pump.fun API, PumpPortal API, DexScreener, fal.ai, and IPFS
  • Signing Server — local Express server on port 3142 serving wallet-connect pages
  • Agent System — dual-model orchestrator with 6 autonomous strategies

Data Flow

Every transaction follows the same zero-custody path:

  1. MCP client invokes a tool (e.g., pump_trade)
  2. Tool builds an unsigned transaction via PumpPortal API
  3. Transaction is stored in a session with a CSRF token
  4. Local signing page opens at http://localhost:3142/sign/:id
  5. User connects their browser wallet and signs
  6. Signed transaction is broadcast to Solana RPC

Zero-Custody Model

PumpSDK never touches your private keys.

How It Works

The server builds unsigned transactions and opens a local signing page in your browser. Your wallet extension (Phantom, Solflare, etc.) handles the actual signing. The server never has access to your private keys, seed phrase, or wallet credentials.

Security Boundaries

  • CSRF Protection — every signing session gets a unique 64-character hex token validated on POST
  • Localhost Only — the signing server binds to localhost:3142 and rejects cross-origin requests
  • Session Expiry — sessions expire after 10 minutes (configurable via PUMPSDK_SESSION_TTL)
  • No Storage — unsigned transactions live in memory only and are pruned on expiry
Tip The signing page supports 8 wallet adapters: Phantom, Solflare, Backpack, Coinbase Wallet, Trust Wallet, Brave Wallet, Exodus, and OKX.

Creative-director prompt builder for AI-generated token logos.

How It Works

PumpSDK uses fal.ai's Nano Banana 2 Pro model to generate token logos. Instead of passing raw prompts, a creative-director system builds structured prompts with visual consistency across all generated images.

  • Japanese Mon — single-weight brushstroke geometry
  • Pentagram Studio — heavy geometric letterform emblem

The resolveSubject function detects whether a token name references a concrete noun (animal, fire, moon, etc.) or an abstract concept. Concrete names get a geometric glyph; abstract names get bold typographic treatment of the initial letterform.

Orchestrator

Dual-model routing for agent tasks.

Two-Model Pattern

The orchestrator routes tasks to one of two models based on complexity analysis:

Hermes 4 (Nous Research)

Fast routing model for routine operations: balance checks, fee claims, price quotes, listing feeds, status monitoring. Low latency, high throughput.
Triggers: claim, balance, check, list, monitor, status, positions, feed, quote, price, holdings

Claude Sonnet (Anthropic)

Strategic analysis model for complex decisions: evaluations, recommendations, comparisons, graduation analysis, sniping criteria, scout ideation.
Triggers: evaluate, analyze, recommend, optimize, compare, strategy, report, graduation, snipe, scout

Routing Logic

The routeDecision function checks the task description against two pattern lists. Strategic patterns take priority; if no strategic pattern matches, routine patterns are checked. Default fallback is Hermes for maximum speed.

Tip Use PUMPSDK_AGENT_DRY_RUN=1 to test routing decisions without executing real transactions.

Strategies

6 autonomous strategies for the agent system.

auto-claim

Periodically scan for unclaimed creator fees above a configurable threshold and build claim transactions. Runs on an interval loop.
Config: wallet, threshold (SOL), interval (ms)

fee-optimizer

One-shot Sonnet-powered fee configuration analysis. Reads current fee splits and recommends optimal configurations based on token performance data.
Config: mint, wallet

graduation-watch

Alert when tracked tokens approach bonding curve completion. Monitors reserve ratios and notifies when graduation is imminent.
Config: mints[], threshold (percentage), interval (ms)

launch-monitor

WebSocket-driven instant alerts for new token creations matching configurable criteria. Filters by name patterns, creator wallets, or market cap thresholds.
Config: filters, alertCallback

sniper

Auto-detect new tokens matching criteria and propose buy transactions. Builds unsigned swaps for tokens meeting market cap, holder count, and bonding curve filters.
Config: criteria, maxBuy (SOL), slippage

scout

Scan trending topics and generate complete token launch packages. Uses Sonnet for creative ideation and builds ready-to-launch bundles with metadata and images.
Config: theme, count, autoLaunch
Danger The sniper and auto-claim strategies can build real transactions. Always test with PUMPSDK_AGENT_DRY_RUN=1 before enabling live mode.

Create and launch a new Pump.fun token step by step.

User
Launch a new token called "Cosmic Cat" with symbol CCAT. Description: "A feline explorer of the Solana cosmos."
Assistant

Assistant
Your launch page is ready at http://localhost:3142/launch/abc123. Connect your wallet, set your dev buy amount, and sign to create the token on-chain.

What Happens Behind the Scenes

  1. pump_launch generates a token logo via fal.ai Nano Banana 2 Pro
  2. Image and metadata JSON are uploaded to IPFS via Pump.fun's pinning service
  3. An ephemeral keypair is generated for the new token mint
  4. The multi-phase launch page opens with pre-filled metadata
  5. User connects wallet, adjusts dev buy slider, and signs the transaction bundle
  6. Transactions are broadcast: create token, set fee config, and execute dev buy
Warning Token launches are on mainnet by default. PumpPortal does not support devnet for launch transactions.

HTTP Transport

Connect to PumpSDK over HTTP/SSE instead of stdio.

When to Use HTTP

Use HTTP transport when your MCP client runs in a different process, container, or machine. The server exposes an SSE endpoint for server-to-client events and a POST endpoint for client-to-server messages.

Configuration

json{
  "mcpServers": {
    "pumpsdk": {
      "transport": "http",
      "url": "http://localhost:3000/mcp"
    }
  }
}
Tip The signing server always runs on port 3142 regardless of transport. HTTP transport only changes how the MCP protocol messages are delivered.

Troubleshooting

Common issues and their solutions.

Session Keypair Expired

Error: Session keypair expired

The signing session has exceeded its TTL (default: 10 minutes). Restart the operation to create a fresh session. Increase the timeout with PUMPSDK_SESSION_TTL if needed.

PumpPortal API Error: HTTP 400

Error: PumpPortal API error: HTTP 400

PumpPortal is mainnet only. If you are running with PUMPSDK_DEVNET=1, trade and launch operations will fail. The signing server UI works on devnet, but transaction building through PumpPortal requires mainnet.

fal.ai Timeout

Error: fal.ai Nano Banana failed: HTTP 504

Check that FAL_API_KEY is set and valid. Verify network connectivity to fal.run. Image generation typically completes in 2-5 seconds; timeouts suggest network issues.

CSRF 403 Errors

Error: 403 Forbidden on signing page POST

The signing server validates that requests originate from http://localhost:3142. Cross-origin requests are rejected. Do not proxy the signing server or access it from a different hostname. For development, set PUMPSDK_DEV_NO_CSRF=1.

No Solana Wallets Detected

The signing page could not find any wallet adapters. Install at least one supported wallet extension: Phantom, Solflare, Backpack, Coinbase Wallet, Trust Wallet, Brave Wallet, Exodus, or OKX.

Build Errors

Run npm run build and check for TypeScript compilation errors. The build step also copies page.html, scout-page.html, and solana-web3.js into the dist directory. Missing files indicate an incomplete build.

Session TTL Too Short

If signing sessions expire before you can complete the flow, increase the TTL:

bashPUMPSDK_SESSION_TTL=1200000 npx pumpsdk   # 20 minutes

FAQ

Frequently asked questions about PumpSDK.

Does PumpSDK store my private keys?

No. PumpSDK uses a zero-custody model. Unsigned transactions are built server-side and signed in your browser wallet. No private keys, seed phrases, or wallet credentials are ever transmitted to or stored by the server.

Can I use devnet?

Partially. The signing server UI works on devnet for testing wallet connectivity and signing flows. However, PumpPortal's trade and launch APIs are mainnet only. Set PUMPSDK_DEVNET=1 to enable devnet mode — mainnet-only tools will return descriptive errors.

What wallets are supported?

The signing pages support 8 wallet adapters:

  • Phantom
  • Solflare
  • Backpack
  • Coinbase Wallet
  • Trust Wallet
  • Brave Wallet
  • Exodus
  • OKX Wallet

How does image generation work?

PumpSDK uses fal.ai's Nano Banana 2 Pro model. A creative-director prompt builder constructs structured prompts with deterministic style lineages and palette pairs based on the token name and symbol. The result is a circular emblem optimized for thumbnail display at 32px.

Can I run this headless?

Yes. PumpSDK communicates via stdio by default, which works with any MCP client. The signing server still needs a browser for wallet interaction, but all non-signing tools work fully headless. For automated workflows, use the agent system with strategies that handle signing programmatically.
